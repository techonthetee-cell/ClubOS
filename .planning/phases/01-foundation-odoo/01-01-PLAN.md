  <name>Task 1: Deploy Odoo 18 + PostgreSQL via Docker Compose</name>
  <files>odoo/docker/docker-compose.yml, odoo/docker/.env, odoo/config/odoo.conf</files>
  <action>
Create the Odoo Docker infrastructure at the project root (NOT inside an app — this lives alongside the monorepo):

1. Create `odoo/docker/docker-compose.yml` with:
   - `odoo:18` official image on port 8069
   - `postgres:16-alpine` with healthcheck (`pg_isready -U odoo`)
   - Named volumes: `odoo-data`, `postgres-data`
   - Odoo depends_on postgres with `condition: service_healthy`
   - Odoo volumes: `odoo-data:/var/lib/odoo`, `../addons:/mnt/extra-addons`, `../config:/etc/odoo`
   - Odoo env: HOST=postgres, USER=odoo, PASSWORD from .env
   - Postgres env: POSTGRES_USER=odoo, POSTGRES_PASSWORD from .env, POSTGRES_DB=postgres
   - Postgres port bound to 127.0.0.1:5432 only (not exposed externally)

2. Create `odoo/docker/.env` with:
   - ODOO_DB_PASSWORD=odoo (dev default, gitignored in production)

3. Create `odoo/config/odoo.conf` with:
   - admin_passwd = clubos-master-pwd (for database manager)
   - db_host = postgres, db_port = 5432, db_user = odoo, db_password = odoo
   - dbfilter = .* (allow all databases — Next.js controls routing, not Odoo)
   - list_db = True (enable multi-database listing for Phase 1 dev)
   - addons_path = /mnt/extra-addons,/usr/lib/python3/dist-packages/odoo/addons

4. Create `odoo/addons/.gitkeep` (empty dir for custom modules later)

5. Start Docker Compose, wait for healthy, then create the first database:
   - `docker compose -f odoo/docker/docker-compose.yml up -d`
   - Wait for postgres healthcheck to pass
   - Access Odoo at http://localhost:8069 — it will show the database manager
   - Create database `clubos_demo` via the Odoo web UI at `/web/database/manager` OR via CLI:
     `docker compose -f odoo/docker/docker-compose.yml exec odoo odoo -d clubos_demo -i base --stop-after-init`
   - Verify Odoo is responding: `curl -s http://localhost:8069/web/database/list | head`

IMPORTANT: This is a Windows dev machine. Use Docker Desktop for Windows. Paths use forward slashes in docker-compose.yml. If volume mount issues occur, check WSL2 backend is enabled.
  </action>

  <action>
NOTE: This package lives at the project root as `packages/odoo-client/` for now. It will be integrated into the Turborepo monorepo in Plan 02. For now, it's a standalone TypeScript package.

1. Create `packages/odoo-client/package.json`:
   - name: "@clubos/odoo-client"
   - version: "0.1.0"
   - main: "dist/index.js", types: "dist/index.d.ts"
   - dependencies: @rlizana/odoo-rpc
   - devDependencies: typescript, vitest, @types/node
   - scripts: build (tsc), test (vitest run), test:unit (vitest run tests/unit), test:integration (vitest run tests/integration)

2. Create `packages/odoo-client/tsconfig.json`:
   - target: ES2022, module: Node16, strict: true
   - outDir: dist, rootDir: src, declaration: true

3. Create `packages/odoo-client/src/types.ts`:
   - OdooConfig: { url: string; db: string }
   - OdooSearchReadOptions: { limit?: number; offset?: number; order?: string }
   - OdooDomain: Array of [string, string, any] tuples (Odoo domain filter format)

4. Create `packages/odoo-client/src/errors.ts`:
   - OdooAuthError extends Error (authentication failures)
   - OdooRPCError extends Error (JSON-RPC errors from Odoo, include fault code)
   - OdooConnectionError extends Error (network/timeout errors)

5. Create `packages/odoo-client/src/auth.ts`:
   - createServiceClient(config: OdooConfig, apiKeyOrPassword: string): authenticates and returns ClubOSClient
   - This is the pattern for server-side usage — re-authenticates per request (serverless safe per Pitfall 1 in research)

6. Create `packages/odoo-client/src/client.ts` — ClubOSClient class:
   - Constructor: takes OdooConfig, creates OdooRPC instance
   - authenticate(login: string, password: string): Promise<number> — returns uid
   - searchRead<T>(model: string, domain: OdooDomain, fields: string[], options?: OdooSearchReadOptions): Promise<T[]>
   - create(model: string, values: Record<string, any>): Promise<number>
   - write(model: string, ids: number[], values: Record<string, any>): Promise<boolean>
   - unlink(model: string, ids: number[]): Promise<boolean>
   - callMethod(model: string, method: string, args: any[], kwargs?: Record<string, any>): Promise<any>
   - Private ensureAuth() throws OdooAuthError if not authenticated
   - Wrap all RPC calls in try/catch, convert to typed errors

7. Create `packages/odoo-client/src/models/base.ts`:
   - BaseModel class: generic typed model operations
   - Constructor takes client + model name
   - Typed search, read, create, update, delete methods
   - This is the pattern for per-model wrappers (partner.ts, etc. in later phases)

8. Create `packages/odoo-client/src/index.ts`:
   - Re-export ClubOSClient, all types, all errors, createServiceClient

9. Write unit tests (mock @rlizana/odoo-rpc):
   - Test constructor, ensureAuth, error wrapping, method delegation

10. Write integration tests (require running Odoo Docker):
   - Test authenticate, CRUD cycle on res.partner
   - Use env vars ODOO_URL=http://localhost:8069, ODOO_DB=clubos_demo, ODOO_USER=admin, ODOO_PASSWORD=admin

11. Create `scripts/test-odoo-connection.ts`:
   - Quick standalone script that authenticates to Odoo and reads res.partner
   - Useful for manual verification: `npx tsx scripts/test-odoo-connection.ts`
